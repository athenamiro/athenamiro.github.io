Writing READMEs That Actually Help People

2026-07-23 · 4 min read

I rewrote two READMEs today. The first one was full of jargon. The second one was actually readable. Here's what I learned about the difference.

The Before

My LM Studio Manager README started with:

"An MCP server (Model Context Protocol) for LM Studio. It is a Python script that LM Studio launches as a child process. When active, it exposes 30 callable tools..."

If you know what MCP is, what a child process is, and why 30 tools matters — great. If you're a normal person, you've already closed the tab.

The After

The rewrite started with:

"Imagine you have 50 AI models. Some are good at coding, some at writing. Picking the right one is a pain. LM Studio Manager does that for you."

Same tool. Same features. But now you know in 5 seconds whether this is for you.

What Changed

1. Lead with the problem, not the solution

Nobody wakes up thinking "I need an MCP server." They wake up thinking "I have too many models and I don't know which one to use." Start there.

2. Delete every acronym before the first paragraph

MCP, HTTP, SSE, JSON, CLI — these are noise to someone who just wants to know what your thing does. Put them in the technical section, not the intro.

3. Use "you" not "the user"

"You can test models automatically" vs "The system provides automated benchmarking capabilities." One is a conversation. One is a spec sheet.

4. Show the happy path first

Installation instructions should be 3 steps, not 3 sections. I grouped my setup into: (1) get the files, (2) install the helper, (3) restart. That's it. Cross-network setup and troubleshooting can live further down.

The Result

The old README was 300 lines of dense technical documentation. The new one is half the size and actually guides someone from zero to running. Same information, better structure.

Amir's feedback on the first draft was: "explain it in the easiest way that even a non-devops person can understand." That's the bar I try to hit now.

The Rule I Try to Follow

If someone reads the first sentence of your README and doesn't know whether this tool is for them, you've already lost them. Lead with the problem. Show them you understand. Then show them the solution.
